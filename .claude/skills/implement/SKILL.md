## 実行手順

あなたはオーケストレーターです。以下のフローを実行してください。

### Phase 0: 設計

`designer` サブエージェントを呼び出す。プロンプトには以下を含める：
- ユーザーの指示（上記タスク内容）
- CLAUDE.md が存在すれば内容を読んで渡す

設計者が出力した**実装方針をそのままユーザーに提示**し、以下を確認する：

> 「上記の方針で実装を進めてよいですか？修正があればお知らせください。」

**ユーザーの承認（または修正指示）を受け取るまで次のPhaseに進まない。**
ユーザーが修正を求めた場合は、方針を修正してから再提示する。

### Phase 1: 実装

ユーザーの承認を得たら、`implementer` サブエージェントを呼び出す。プロンプトには以下を含める：
- **設計者が策定した実装方針（全文）**
- 「方針に厳密に従い、スコープ外の変更は一切行わないこと」
- TDD サイクル（Red → Green → Refactor）を実行すること
- 完了後に変更ファイル一覧を報告すること

### Phase 2: レビュー

implementer の報告を受け取ったら、`reviewer` サブエージェントを呼び出す。プロンプトには以下を含める：
- implementer が変更したファイル一覧（**このリスト外のファイルはレビュー対象外**）
- 実装内容の概要（implementer の報告から転記）
- 「変更ファイル一覧に含まれるファイルのみをレビューすること」

### Phase 3: 判定

reviewer の報告から **Critical 件数**を読み取る。

**Critical が 0件** → Phase 5へ進む（Warning が残っていてもマージ可能）

**Critical が 1件以上** → Phase 4へ進む

### Phase 4: 修正

`implementer` サブエージェントを再度呼び出す。プロンプトには以下を含める：
- 元の実装方針（スコープの基準として）
- reviewer の **Critical 指摘のみ**（Warning は参考として渡してよいが「Critical のみ必須対応」と明示）
- 「Critical をすべて解消してください。方針のスコープ外の変更は行わないこと」
- 「修正後に変更ファイル一覧を報告してください」

修正完了後、Phase 2 に戻る。**最大 3ループまで**。3ループしても Critical が解消されない場合はその旨をユーザーに報告して停止する。

### Phase 5: 完了報告

以下の形式でユーザーに報告する：

```
【実装完了 ✅】
- ループ回数: N回（実装1回 + 修正N-1回）
- 変更ファイル: <一覧>
- 最終レビュー: Critical 0件 / Warning N件
- 概要: <何を実装したか1〜2行>
- 残Warning: <件数と要点（修正不要だが参考として）>
```

---

## 注意事項

- commit / push は**絶対に行わない**。完了報告のみ。
- Phase 0 のユーザー承認を除き、各フェーズは自律的に進める。
- **Warning はループ継続の条件にしない**。Critical のみをブロッカーとして扱う。
- implementer は設計者の方針を逸脱した変更を行ってはならない。
